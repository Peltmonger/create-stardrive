# create-stardrive

CLI launchpad that scaffolds a new project from the [Astro Stardrive](https://github.com/Peltmonger/stardrive) boilerplate. Run via `npm create stardrive@latest` (or the pnpm/yarn/bun equivalents).

## What it does (in order)

1. `parseArgs` — reads `--no-install`, `--version <x.y.z>` / `-v`, and one positional project name.
2. `detectPackageManager` — sniffs `npm_config_user_agent` to pick npm/pnpm/yarn/bun, then verifies the binary exists.
3. `askProjectName` — uses the positional arg if valid and the dir is free; otherwise prompts interactively (default `my-stardrive`). Bails if stdin isn't a TTY.
4. `resolveTag` — `git ls-remote --tags` against the Stardrive repo, picks the latest stable `vX.Y.Z` (or the requested version, normalized with a leading `v`).
5. `cloneRepo` — shallow clone the tag into a tmp dir, strip `.git`, copy to `targetDir`. Uses `-c advice.detachedHead=false` to silence the detached-HEAD warning.
6. `trimProject` — removes `scripts/syncVersion.js`, `SECURITY.md`, `.github`, and any stale lockfiles.
7. `calibratePackageJson` — sets `name` to the project name, drops `sync-version` and `prebuild` scripts, sets `packageManager` for pnpm/bun.
8. `installDependencies` — runs `<pm> install` (skipped with `--no-install`).

## Source layout

Source lives in `src/`; `bin/index.js` is generated by esbuild and **gitignored**.

| File | Role |
| --- | --- |
| `src/index.ts` | Top-level orchestration — the script the CLI runs. |
| `src/args.ts` | Tiny argv parser. No deps. |
| `src/logger.ts` | ANSI color helpers + `step`/`info`/`ok`/`fail` + banner. Honors `NO_COLOR` and non-TTY. |
| `src/package-manager.ts` | PM detection, `commandExists`, `devCommands` map. |
| `src/project-name.ts` | Validation regex + interactive prompt loop. |
| `src/release.ts` | `git ls-remote` tag listing + semver compare + tag resolution. |
| `src/scaffold.ts` | Clone, trim, calibrate package.json, install. |
| `src/types.ts` | `PackageManager` + `CliArgs`. |
| `build.mjs` | esbuild bundle → `bin/index.js`, prepends `#!/usr/bin/env node`, chmod +x. |

## Build & publish

- `npm run build` → esbuild bundles `src/index.ts` to `bin/index.js` (ESM, node18 target, shebang banner, executable bit).
- `npm run typecheck` → `tsc --noEmit` against the strict config.
- `package.json` `files: ["bin"]` — only the bundled artifact ships.
- `.github/workflows/npm-publish.yml` runs on a published GitHub Release: validates `vX.Y.Z` tag pattern, that it's based on `main`, that the tag matches `package.json` version, then `npm ci && npm run build && npm publish` (npm trusted-publishers via OIDC, no token).

## TypeScript config notes

- `strict`, `noUncheckedIndexedAccess`, `noImplicitOverride`, `verbatimModuleSyntax`, `noEmit`.
- `module: NodeNext` — **imports must use `.js` extensions** (e.g. `./logger.js`), even though source is `.ts`. esbuild handles it; tsc requires it.
- `isolatedModules` + `verbatimModuleSyntax` — type-only imports need `import type`.

## Session learnings (Jun 2026 — TS rewrite)

- The published artifact stayed behaviorally identical. The rewrite was purely a structural split: one ~345-line `bin/index.js` → eight typed modules + an esbuild bundle.
- `bin/index.js` was deleted from git tracking and added to `.gitignore`. The build step in the publish workflow regenerates it before `npm publish` reads `files: ["bin"]`.
- The previous `files` entry was `["index.js"]`; this changed to `["bin"]` along with the new `bin/` output location. Don't regress this — npm will publish nothing useful otherwise.
- No `prepublishOnly` script was added in the end; the CI workflow runs `npm run build` explicitly before `npm publish`. If you ever publish manually from a laptop, remember to build first.
- The PR's CLAUDE.md was apparently not included in the merge — hence this file. Commit it directly to `main` next time.

## Conventions worth keeping

- No runtime deps. The CLI relies only on Node built-ins + `git` on PATH. Keep it that way; users run this with `npm create` and dep-installs there are painful.
- All user-visible logging goes through `logger.ts` so color/TTY handling stays consistent.
- Project name validation: `/^[a-z0-9._-]+$/i` and may not start with `.` or `_`. Mirror this if you add more entry points.
- Semver comparison in `release.ts` is hand-rolled (3-part numeric, strips `-prerelease`). Don't pull in a semver lib for this — the input is `git ls-remote` output we control.
